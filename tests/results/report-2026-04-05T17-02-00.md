### Gemini Rate Limit Analysis

The PDF queue (`pdfQueue.ts`) is **client-side only** — it serializes uploads
per browser tab but does NOT coordinate across multiple concurrent users.

**Impact:** With 20 users each uploading 3 PDFs simultaneously:
- The server receives up to 60 concurrent `/api/extract-policy` requests
- Gemini Free Tier: **15 RPM hard limit** → 429 errors expected for 45+ requests
- Each 429 triggers a 10s client-side retry (`RETRY_DELAY_MS = 10000`)
- **No server-side queue** means all 60 requests hit Gemini simultaneously

**Recommendation:** Move the queue to a server-side Redis/BullMQ worker or
use Gemini's built-in streaming with a shared per-minute counter in MongoDB.

### Architecture Improvements

1. **Server-Side PDF Queue** — Use `pg-boss` or `BullMQ` to process extractions
   asynchronously. Return a `jobId` immediately and poll for results. This
   eliminates the 10s serverless timeout risk.

2. **Gemini Quota Pooling** — Rotate between multiple free Gemini API keys
   (one per GCP project) to multiply the effective RPM limit.

3. **Connection Pooling** — Upgrade from `connectDB()` per-request to a
   proper singleton with `maxPoolSize` set. Mongoose already does this if
   `MONGODB_URI` is cached — verify `global._mongooseCache` is being used.

4. **Rate Limit Headers** — Add `Retry-After` headers to 429 responses so
   the client queue can back off intelligently instead of using a fixed 10s.

5. **Vercel Edge Caching** — Cache dashboard stats for 30s with
   `res.setHeader("Cache-Control", "s-maxage=30, stale-while-revalidate=60")`
   to reduce MongoDB load under concurrent dashboard views.
